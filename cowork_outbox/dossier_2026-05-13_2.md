# Phantom Candidate "2" — Dossier NOT Built

**Brief id:** `dossier_2026-05-13_2`
**Status:** ERROR — phantom candidate row in the registry.

## What happened

The Windows-side scanner produced a `candidate_dossier` brief for a candidate whose `name` is the literal string `"2"` and whose `office`, `party`, `district`, and event dates are all empty. The `known_events` array contains seven Maryland House Bills (HB0001 through HB0008 of the 2026 Regular Session) all of whose `title` is also the literal string `"2"`.

This is a data-quality bug in the upstream extraction pipeline, not a real candidate.

## Why no dossier was produced

There is no person to research. A web search on "candidate '2'" returns nothing useful and would otherwise risk fabrication. Per the task hard rule — "Never fabricate vote counts, bill numbers, quotes, school names, employer names, family details, or dates. Real URLs only." — the only correct output is this error stub.

## What is probably going on (root cause hypothesis)

The data extraction step that populates `politicians.name` from raw HTML/PDF sources appears to be writing a column ordinal (or HB number) into the `name` column for at least one row, producing a politician named "2" whose linked events are all 2026 House Bills with sequential numbers. Three possible failure modes:

1. A column-index mix-up in the Maryland General Assembly bill-detail parser — the bill number column is being mistakenly written to the politician's name column.
2. A NULL-coalesce fallback that converted a missing politician name to the string representation of an integer foreign key.
3. A test/seed row that was never cleaned out before the scanner ran in production mode.

## Recommended remediation

1. Look at the row in `politicians` with `name = '2'` in `politics.db`. (Cannot be read from the drain side this run — SQLite version mismatch — but is readable on the Windows side.)
2. Find the source pipeline that inserted the row and fix the upstream extraction.
3. Delete the phantom row OR re-link the seven listed bills to their actual sponsor(s).
4. Add a guard in `main.py series cowork-queue` that refuses to emit a `candidate_dossier` brief when `name` is numeric-only or shorter than 3 characters.

## What is in the registry that may be related

The registry file `data/candidate_series.json` is itself malformed (truncated mid-record at the Brooke Lierman row, line ~813). The phantom "2" candidate may have been introduced when an interrupted write corrupted the registry — repairing the registry from git history should be the first move, since it is likely the upstream cause of the phantom row.
